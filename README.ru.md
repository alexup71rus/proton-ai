# Proton AI

Proton AI - конструктор небольших локальных моделей с поддержкой tool calling. Он дает готовую среду для разработки и тестирования. Поведение задается датасетом: это может быть маршрутизатор вызовов инструментов, ИИ для любой игры, небольшая диалоговая модель, классификатор или другой специализированный локальный сценарий.

Документация на английском: [README.md](README.md)

## Встроенный сценарий вызова инструментов

```text
реестр инструментов + запрос пользователя -> JSON tool_calls
```

Обычный вызов инструмента:

```json
{"tool_calls":[{"name":"get_current_time","arguments":{}}]}
```

Fallback тоже возвращается как структурированный вызов:

```json
{"tool_calls":[{"name":"__fallback__","arguments":{}}]}
```

Общий цикл продукта:

```text
описать задачу -> собрать датасет -> обучить модель -> проверить -> изучить логи -> улучшить датасет
```

## Для чего это нужно

Небольшим локальным моделям часто нужна одна и та же инфраструктура: формат датасета, команды обучения, управление артефактами, экран проверки, логи и способ улучшать следующий датасет. Proton AI собирает этот цикл в локальный конструктор, чтобы не начинать каждый эксперимент с нуля.

Встроенный пример с вызовом инструментов полезен там, где нужен структурированный результат, а не свободный текст:

- локальные команды автоматизации;
- выбор операций внутреннего API;
- классификация по разрешенным действиям;
- запуск скриптов с проверенными аргументами;
- маленькие офлайн-маршрутизаторы под фиксированный набор инструментов.

## Архитектура

```text
service/      FastAPI-сервис модели: маршрутизация, проверка, генерация датасета, обучение
web_backend/  FastAPI-бэкенд интерфейса: настройки, инструменты, датасеты, выполнение, логи
web_ui/       веб-интерфейс на React/Vite
data/         локальные инструменты, датасеты, веса, токенизаторы, логи, состояние проекта
repo_docs/    документация проекта на английском и русском
```

Внутренний Python-пакет пока называется `protonx`. Публичное имя проекта - `Proton AI`, имя репозитория - `proton-ai`.

## Требования

- Python 3.11 или 3.12
- Node.js 18+ и npm
- macOS, Linux или другая среда, поддерживаемая PyTorch

Рекомендуется виртуальное окружение:

```bash
python3.11 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -r service/requirements.txt
python -m pip install -r web_backend/requirements.txt
python -m pip install -r requirements-dev.txt
(cd web_ui && npm install)
```

Если используется другая команда Python, используй один и тот же интерпретатор для установки зависимостей, тестов и `uvicorn`.

## Локальный запуск

Все процессы:

```bash
make run-dev
```

Отдельно:

```bash
make run-service
make run-ui-backend
make run-web-ui
```

Адреса:

- `http://127.0.0.1:8000/health` - сервис модели
- `http://127.0.0.1:8100/health` - бэкенд веб-интерфейса
- `http://localhost:8501` - веб-интерфейс

Сейчас веб-интерфейс включает страницы для встроенного сценария вызова инструментов:

- **Tools** - редактирование реестра инструментов и путей к исполняемым сценариям.
- **Training** - импорт или генерация датасетов, проверка и запуск обучения.
- **Test** - проверка запроса пользователя на выбранной модели, результат проверки и выполнения.
- **Logs** - просмотр проблемных маршрутизаций и экспорт неудачных примеров в черновик датасета.

## Как собрать примерный маршрутизатор

1. Добавь инструменты на странице **Tools** или в `data/tools/tools.json`.
2. Укажи безопасные и доверенные `executor_path`.
3. Открой **Training** и выбери папку для датасетов.
4. Импортируй компактный JSONL-датасет или сгенерируй стартовый датасет из реестра инструментов.
5. Запусти обучение новой модели.
6. Проверь реальные запросы на странице **Test**.
7. Используй **Logs**, чтобы найти недостающие фразы, синонимы, значения аргументов и случаи, где нужен fallback.

Компактная JSONL-строка:

```json
{
  "tools": [
    {"name":"get_current_time","tags":["time","date"]},
    {"name":"__fallback__","tags":["fallback","no tool"]}
  ],
  "user": "what time is it",
  "assistant": {
    "tool_calls": [{"name":"get_current_time","arguments":{}}]
  }
}
```

## Переменные окружения

Публичные имена используют `PROTON_AI_*`. Старые `PROTONX_*` имена пока поддерживаются для совместимости.

| Переменная | Назначение | По умолчанию |
| --- | --- | --- |
| `PROTON_AI_TOOLS_FILE` | файл реестра инструментов | `data/tools/tools.json` |
| `PROTON_AI_DATASET_DIR` | папка для датасетов | `data/train/routing` |
| `PROTON_AI_ROUTER_LOG_FILE` | JSONL-файл логов маршрутизатора | `data/logs/router.jsonl` |
| `PROTON_AI_WORKSPACE_FILE` | файл настроек веб-интерфейса | `data/workspace/settings.json` |
| `PROTON_AI_SERVICE_URL` | URL сервиса модели для бэкенда интерфейса | `http://127.0.0.1:8000` |
| `PROTON_AI_TRAIN_DEVICE` | устройство для обучения | `cpu` |
| `PROTON_AI_TRAIN_STATE_PATH` | файл состояния обучения | `data/workspace/training_state.json` |

## Проверка

```bash
python -m pytest --import-mode=importlib service/tests web_backend/tests -q
(cd web_ui && npm run build)
```

Длинные тесты синтетического датасета полезны перед изменениями генератора, но не обязательны для обычных правок интерфейса и бэкенда.

## Перед публикацией

Сгенерированное состояние не должно попадать в git:

- `data/train/*`
- `data/weights/*`
- `data/tokenizers/*`
- `data/tools/*`
- `data/logs/*`
- `data/workspace/settings.json`

В git остаются примеры и `.gitkeep`, чтобы структура локальных папок была понятна.
